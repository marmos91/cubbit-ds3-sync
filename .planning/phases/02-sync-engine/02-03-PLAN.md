**1. Create S3LibListingAdapter.swift in DS3DriveProvider/:**

This adapter bridges the existing S3Lib (which lives in DS3DriveProvider, not DS3Lib) to the SyncEngine's `S3ListingProvider` protocol. The adapter must be in the extension target (DS3DriveProvider) since it depends on S3Lib which is only in that target.

```swift
import Foundation
import DS3Lib

/// Adapts the existing S3Lib to conform to SyncEngine's S3ListingProvider protocol.
/// Lives in the extension target because S3Lib is only available there.
final class S3LibListingAdapter: S3ListingProvider, @unchecked Sendable {
    private let s3Lib: S3Lib
    private let drive: DS3Drive

    init(s3Lib: S3Lib, drive: DS3Drive) {
        self.s3Lib = s3Lib
        self.drive = drive
    }

    func listAllItems(bucket: String, prefix: String?) async throws -> [String: S3ObjectInfo] {
        var allItems: [String: S3ObjectInfo] = [:]
        var continuationToken: String? = nil

        repeat {
            let (items, nextToken) = try await s3Lib.listS3Items(
                forDrive: drive,
                withPrefix: prefix,
                recursively: true,
                withContinuationToken: continuationToken
            )

            for item in items {
                let key = item.itemIdentifier.rawValue
                allItems[key] = S3ObjectInfo(
                    etag: item.metadata.etag,
                    lastModified: item.metadata.lastModified,
                    size: Int64(truncating: item.metadata.size),
                    contentType: item.metadata.contentType,
                    parentKey: item.parentItemIdentifier == .rootContainer ? nil : item.parentItemIdentifier.rawValue,
                    isFolder: item.isFolder
                )
            }

            continuationToken = nextToken
        } while continuationToken != nil

        return allItems
    }
}
```

**3. Add pinning support to S3Item:**

Per Apple docs: When the user pins a file, the system calls `modifyItem` with `.contentPolicy` in changedFields. The system manages pinning state on its side. Our extension acknowledges the change.

In FileProviderExtension.modifyItem, add handling for content policy changes:
```swift
if changedFields.contains(.contentPolicy) {
    // System is telling us the user pinned/unpinned this item
    // The system tracks pinning state -- just acknowledge the change
    cb.handler(s3Item, NSFileProviderItemFields(), false, nil)
    return progress
}
```

For S3Item, add support for returning the correct content policy when the system provides it. When creating S3Items from MetadataStore data for enumeration, check `isMaterialized`:
- If a file was explicitly kept downloaded (system manages this), return `.downloadLazilyAndKeepDownloaded`
- Default: return `.downloadLazily` (cloud placeholder)

Since the system manages pin state, the simplest correct approach: always return `.downloadLazily`. The system overrides for pinned items. But to be explicit, add an `isPinned: Bool = false` init parameter to S3Item:

```swift
private let isPinned: Bool

var contentPolicy: NSFileProviderContentPolicy {
    isPinned ? .downloadLazilyAndKeepDownloaded : .downloadLazily
}
```

In enumerateChanges, when creating S3Items from ReconciliationResult, pass `isPinned: false` (default). The system will override if the user has pinned the item.
